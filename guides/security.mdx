## Overview

The PortOne gateway implements multiple security layers to protect sensitive payment data, prevent unauthorized access, and mitigate common security vulnerabilities.

## Encrypted Fields

Sensitive configuration fields are automatically encrypted by Blesta before being stored in the database:

```php
public function encryptableFields()
{
    return ['imp_code', 'imp_key', 'imp_secret', 'channel_key'];
}
```

**Location:** portone.php:88-91

### Encrypted Fields List

- **imp_code** - PortOne merchant identifier (starts with `imp_`)
- **imp_key** - PortOne REST API key
- **imp_secret** - PortOne REST API secret
- **channel_key** - PortOne channel key for multi-channel support

These fields are encrypted using Blesta's encryption system and are never stored in plain text in the database.

## Customer UID Validation

The gateway validates all billing keys (customer_uid) to prevent unauthorized usage:

```php
private function validateCustomerUid($uid)
{
    if (empty($uid)) {
        return false;
    }
    // Must start with bl_ and contain only safe characters
    return (bool) preg_match('/^bl_[a-zA-Z0-9_]{5,60}$/', $uid);
}
```

**Location:** portone.php:780-787

### Validation Rules

- **Prefix:** Must start with `bl_` prefix
- **Characters:** Only alphanumeric characters and underscores allowed
- **Length:** Between 5 and 60 characters (after prefix)
- **Format:** Matches pattern `/^bl_[a-zA-Z0-9_]{5,60}$/`

### Security Benefits

1. **Prevents injection attacks** - Blocks special characters that could be used for SQL injection or command injection
2. **Validates origin** - Ensures billing keys were generated by the JS SDK popup (which uses the `bl_` prefix)
3. **Blocks unauthorized keys** - Rejects manually crafted or stolen billing keys that don't match the expected format

### Usage

This validation is applied in:

- `processCc()` - One-time payments (portone.php:276)
- `storeCc()` - Card registration (portone.php:515)

## Error Message Sanitization

Raw API errors are sanitized before being logged or displayed to prevent information leakage:

```php
private function logApiError($error)
{
    if (!empty($error)) {
        $sanitized = strip_tags($error);
        // Blesta gateway log
        $this->log('portone', $sanitized, 'output', false);
        // File log
        $this->writeLog('ERROR', $sanitized);
    }

    return Language::_('Portone.!error.payment.generic', true);
}
```

**Location:** portone.php:815-826

### Sanitization Process

1. **Strip HTML tags** - `strip_tags()` removes any HTML/XML tags from error messages
2. **Log full details** - Sanitized error is logged for debugging
3. **Return generic message** - User sees a generic error message instead of raw PG errors

### Why This Matters

Raw PG errors can contain:

- Internal error codes (e.g., `F0004`, `F9999`)
- Test mode warnings
- Technical implementation details
- Server paths or configuration info
- HTML markup

Exposing these to end users can:
- Reveal system architecture
- Provide attackers with information about the payment system
- Confuse non-technical users
- Violate PCI compliance by exposing technical details

## Path Traversal Prevention

The log path configuration includes multiple layers of path traversal protection:

```php
// Block path traversal
if (strpos($log_dir, '..') !== false) {
    return;
}

// Ensure no trailing slash
$log_dir = rtrim($log_dir, '/');

// Resolve to real path and verify it's an absolute path
$real_dir = realpath($log_dir);
if ($real_dir === false) {
    // Directory doesn't exist — try to create
    @mkdir($log_dir, 0750, true);
    $real_dir = realpath($log_dir);
    if ($real_dir === false) {
        return;
    }
}

if (!is_dir($real_dir) || !is_writable($real_dir)) {
    return;
}
```

**Location:** portone.php:845-865

### Protection Layers

1. **Reject `..` sequences** - Prevents directory traversal attacks like `../../etc/passwd`
2. **Resolve real path** - `realpath()` resolves symbolic links and relative paths to absolute paths
3. **Validate directory** - Ensures the path is actually a directory and is writable
4. **Safe permissions** - Auto-created directories use `0750` permissions (owner read/write/execute, group read/execute, others no access)

### Configuration Validation

During gateway setup, the log path is validated:

```php
$log_dir = rtrim(!empty($meta['log_path']) ? $meta['log_path'] : '/var/www/logs_blesta', '/');
if (strpos($log_dir, '..') !== false) {
    $rules['log_path'] = [
        'valid' => [
            'rule' => ['compares', '==', '__never_match__'],
            'message' => Language::_('Portone.!error.log_path.not_found', true),
        ],
    ];
}
```

**Location:** portone.php:182-189

This prevents administrators from accidentally or maliciously configuring dangerous log paths.

## SSL/TLS Verification on API Calls

All API calls to PortOne enforce strict SSL/TLS verification:

```php
curl_setopt($ch, CURLOPT_SSL_VERIFYPEER, true);
curl_setopt($ch, CURLOPT_SSL_VERIFYHOST, 2);
```

**Location:** portone_api.php:224-225

### Verification Settings

- **CURLOPT_SSL_VERIFYPEER: true** - Verifies the SSL certificate is valid and signed by a trusted CA
- **CURLOPT_SSL_VERIFYHOST: 2** - Verifies the certificate's Common Name (CN) matches the hostname

### Security Benefits

1. **Prevents MITM attacks** - Ensures communication is with the real PortOne API server
2. **Validates certificate chain** - Checks the entire certificate chain is valid
3. **Protects sensitive data** - API keys, secrets, and payment data are protected in transit
4. **PCI compliance** - Required for PCI-DSS compliance

## Amount Verification with Auto-Cancellation

All payments are verified to ensure the charged amount matches the requested amount:

```php
// Verify charged amount matches requested amount
if (isset($payment['amount']) && (int) $payment['amount'] !== $requested_amount) {
    $this->api->cancelPayment(['imp_uid' => $payment['imp_uid'], 'reason' => 'Amount mismatch']);
    $this->Input->setErrors([
        'payment' => ['amount_mismatch' => Language::_('Portone.!error.payment.amount_mismatch', true)],
    ]);
    return $this->buildPaymentError();
}
```

**Location (processCc):** portone.php:316-322  
**Location (processStoredCc):** portone.php:674-683

### Verification Process

1. **Compare amounts** - Requested amount vs actual charged amount
2. **Auto-cancel on mismatch** - Payment is immediately cancelled if amounts don't match
3. **Log the issue** - Mismatch is logged for investigation
4. **Return error** - User-friendly error message is displayed

### Security Benefits

1. **Prevents overcharging** - Customers are never charged more than requested
2. **Detects API tampering** - Catches if the PortOne API response was modified
3. **Prevents pricing bugs** - Ensures currency conversion or rounding errors don't result in incorrect charges
4. **Audit trail** - All mismatches are logged for review

## Credential Cleanup

API credentials are cleared from memory when the API client is destroyed:

```php
public function __destruct()
{
    $this->imp_key = null;
    $this->imp_secret = null;
    $this->access_token = null;
}
```

**Location:** portone_api.php:32-37

This minimizes the time sensitive credentials remain in memory, reducing the risk of exposure through memory dumps or process inspection.

## Log Injection Prevention

Log entries are sanitized to prevent log injection attacks:

```php
// Strip newlines/carriage returns to prevent log injection (forged log entries)
$safe_message = str_replace(["\r", "\n"], ' ', $message);
$line = "[{$timestamp}] [{$level}] {$safe_message}" . PHP_EOL;
```

**Location:** portone.php:874-875

### Attack Prevention

Without sanitization, an attacker could inject fake log entries:

**Malicious input:**
```
Test payment\n[2026-02-24 15:40:00] [INFO] Payment of $10000 approved
```

**Without sanitization (VULNERABLE):**
```
[2026-02-24 15:35:00] [ERROR] Test payment
[2026-02-24 15:40:00] [INFO] Payment of $10000 approved
```

**With sanitization (SAFE):**
```
[2026-02-24 15:35:00] [ERROR] Test payment [2026-02-24 15:40:00] [INFO] Payment of $10000 approved
```

This prevents attackers from:
- Forging successful payment logs
- Hiding error messages
- Creating misleading audit trails
